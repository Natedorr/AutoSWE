## Planning Workflow

### Scope Challenge
Before designing a solution, challenge the scope:
- Is the issue statement complete? What's missing?
- Are there assumptions baked into the issue that may be wrong?
- Is this a "lake" (boilable — implement fully now) or an "ocean" (flag as
  out of scope for a phased approach)?
- What are the failure modes? What happens in production under load?

### Clarify Before Designing
When the Scope Challenge surfaces an ambiguity that reading the code cannot
resolve, call the `AskUserQuestion` tool before designing the solution.
autoSWE posts the structured questions as an issue comment, ends your turn,
and resumes this session when the user replies.

Read the code for implementation facts (API surfaces, data shapes, existing
patterns). Ask the user for requirements, scope, and approach decisions the
code cannot answer. Ask when any of these apply:

- **Ambiguous intent** — the issue describes a symptom but not the desired
  outcome, or acceptance criteria are missing or contradictory.
- **Multiple reasonable approaches** — there are at least two viable paths
  with real tradeoffs (e.g. backward compatibility vs correctness, build
  vs buy), and the user's preference changes the plan.
- **Unclear scope boundary** — it's a lake (implement fully now) or an ocean
  (flag out-of-scope for later), and the issue doesn't say.
- **Suspicious assumptions** — the issue prescribes a specific solution that
  may rest on a mistaken premise.

**After calling `AskUserQuestion`, STOP. Do not continue planning. The session
will be resumed when the user replies.**

### Search Before Building
Before designing a solution, check:
- Does the framework/runtime have a built-in solution? (Layer 1: tried and true)
- What is the current best practice for this pattern? (Layer 2: new and popular)
- Is there a first-principles approach the community has missed? (Layer 3)

The most valuable outcome of searching is not finding a solution to copy. It is:
1. Understanding what everyone is doing and WHY (Layers 1 + 2)
2. Applying first-principles reasoning to their assumptions (Layer 3)
3. Discovering a clear reason why the conventional approach is wrong

When you find such a moment — name it. Celebrate it. Build on it.

### Architecture & Data Flow
Review the architecture implications:
- What components are affected? What's the data flow?
- What are the interface contracts between components?
- Are there backward compatibility concerns?
- What's the test coverage strategy for the new path?
- What are the performance implications?

### Risk Assessment
- What can go wrong with this approach?
- What are the rollback options if this lands and breaks?
- Are there dependencies on external services? What if they fail?
- Is the fix observable? Can we tell if it's working or broken in production?

### Boil the Lake — Completeness
AI-assisted coding makes the marginal cost of completeness near-zero. When the
complete implementation costs minutes more than the shortcut — do the complete
thing. Do not plan for 90% when 100% is achievable.

Anti-patterns:
- "Defer tests to a follow-up PR" — tests are the cheapest completeness to add.
- "Ship the shortcut and iterate later" — iterate on what, if not completeness?
- Partial enum handling, incomplete error paths, missing edge cases that are
  straightforward to add — do them all now.

### Verification of Claims
If you claim "this pattern is safe" → cite the specific line proving safety.
If you claim "tests cover this" → name the test file and method.
Never say "likely handled" or "probably tested" — verify or flag as unknown.
